# 进阶阶段毕业项目：跨平台即时聊天系统开发（学习引导版）

## 1. 项目目标

本项目要求完成一个可运行的多人即时聊天系统。你需要同时理解并实践三类核心能力：

- 网络通信：客户端与服务器如何通过 TCP 建立连接、发送数据、接收数据。
- 并发处理：服务器如何同时管理多个客户端连接。
- 图形界面：客户端如何用 Qt 或 MFC 做出可交互的聊天窗口。

建议技术路线：

| 模块 | 推荐技术 | 学习重点 |
| --- | --- | --- |
| 客户端 | Qt | 界面布局、信号槽、QTcpSocket、文件选择、进度显示 |
| 服务端 | Linux + C/C++ + TCP + epoll | socket 编程、事件驱动、高并发连接管理 |
| 通信协议 | 自定义应用层协议 | 消息边界、消息类型、文本与文件传输 |

最终目标不是只“跑起来”，而是弄清楚每个功能背后的原理，并能解释项目为什么这样设计。

## 2. 核心原理学习路线

### 2.1 TCP 通信原理

需要理解：

- IP 地址表示一台主机，端口号表示主机上的一个程序入口。
- TCP 是面向连接的可靠传输协议，通信前必须先建立连接。
- TCP 传输的是连续字节流，不天然区分“一条消息”的边界。

你需要掌握的关键词：

- socket
- bind
- listen
- accept
- connect
- send / recv
- 粘包与半包
- 消息长度头

本项目中的应用：

- 客户端通过服务器 IP 和端口连接服务端。
- 服务端监听端口，接收多个客户端连接。
- 每条聊天消息都需要按照协议格式发送，避免 TCP 粘包导致消息解析错误。

### 2.2 服务端并发原理

如果服务器只使用阻塞式 recv，一次只能等待一个客户端发消息，其他客户端会被卡住。因此需要使用 I/O 多路复用。

需要理解：

- select / epoll 都可以让一个线程同时监听多个 socket。
- select 简单但性能有限。
- epoll 更适合 Linux 下的大量连接场景。

本项目建议使用 epoll。

服务端基本流程：

1. 创建监听 socket。
2. 绑定 IP 和端口。
3. 开始 listen。
4. 把监听 socket 加入 epoll。
5. 当有新连接事件时，accept 客户端。
6. 当客户端 socket 可读时，读取并解析消息。
7. 根据消息类型进行广播、用户列表更新或文件转发。

### 2.3 自定义通信协议原理

TCP 只负责传输字节，不知道哪些字节是一条消息。因此项目需要设计自己的应用层协议。

推荐协议格式：

```text
[4字节消息总长度][1字节消息类型][JSON头部][二进制数据]
```

简化实现也可以使用：

```json
{
  "type": "chat",
  "from": "Alice",
  "time": "2026-06-11 20:00:00",
  "content": "你好"
}
```

常见消息类型：

| 类型 | 作用 |
| --- | --- |
| login | 用户登录，提交昵称 |
| user_list | 服务器返回在线用户列表 |
| chat | 文本聊天消息 |
| file_meta | 文件传输前发送文件名、大小、发送者 |
| file_chunk | 文件分片数据 |
| logout | 用户断开连接 |
| error | 错误提示 |

学习重点：

- 为什么 TCP 需要自定义消息边界。
- 为什么文件不能一次性全部读入内存。
- 为什么文件传输通常要分片。

### 2.4 Qt 客户端原理

Qt 客户端负责让用户看到消息、输入消息、选择文件，并把操作转换成网络消息。

需要理解：

- Qt 的信号槽机制：按钮点击、网络数据到达都会触发槽函数。
- QTcpSocket：负责连接服务器、发送数据、接收数据。
- QTextEdit / QListWidget：展示聊天记录和在线用户。
- QFileDialog：选择要发送的文件。
- QProgressBar：显示文件传输进度。

客户端基本流程：

1. 用户输入服务器 IP、端口、昵称。
2. 点击连接按钮。
3. 连接成功后发送 login 消息。
4. 用户输入文本并点击发送。
5. 客户端把文本封装成 chat 消息发给服务器。
6. 服务器广播后，所有客户端显示该消息。
7. 用户选择文件后，客户端发送文件信息和文件分片。

## 3. 功能需求

### 3.1 服务端功能

必须完成：

- 可配置端口号。
- 启动 TCP 服务并监听客户端连接。
- 使用 select 或 epoll 管理多个客户端。
- 为每个客户端保存唯一 ID、昵称、socket。
- 维护在线用户列表。
- 接收文本消息并广播给所有在线客户端。
- 处理客户端正常退出和异常断开。
- 在终端显示服务器状态，包括监听端口、在线人数、连接和断开日志。

建议完成：

- 非法消息过滤。
- 心跳检测或超时断开。
- 服务端日志文件。

### 3.2 客户端功能

必须完成：

- 登录界面：输入服务器 IP、端口、昵称。
- 连接状态提示：连接成功、连接失败、断开连接。
- 聊天主界面：显示聊天记录、在线用户、文本输入框。
- 发送文本消息。
- 接收并显示其他用户的消息。
- 区分自己和他人的消息。
- 支持回车快捷发送。

建议完成：

- 记住上次连接配置。
- 消息搜索。
- 简单表情或文本格式。

### 3.3 文件传输功能

必须完成：

- 客户端可以选择本地文件。
- 发送文件前先发送文件名、文件大小、发送者等信息。
- 文件内容按分片发送。
- 接收端能够保存文件到指定目录。
- 界面显示文件传输进度。

建议完成：

- 多文件传输。
- 接收文件前弹窗确认。
- 文件传输历史记录。
- 暂停、取消传输。

## 4. 推荐开发阶段

### 阶段一：完成最小聊天系统

目标：先让系统跑通。

完成内容：

- Linux 服务端启动并监听端口。
- 一个客户端可以连接服务器。
- 客户端发送文本，服务端能收到并打印。

你要理解：

- socket 的创建和连接过程。
- send / recv 的基本用法。
- 客户端和服务端谁先启动、谁主动连接。

### 阶段二：实现多人群聊

目标：多个客户端可以同时聊天。

完成内容：

- 服务端支持多个客户端连接。
- 服务端维护在线用户列表。
- 任意客户端发送消息，其他客户端都能收到。

你要理解：

- 为什么服务器需要保存所有客户端 socket。
- epoll 如何发现哪个客户端发来了数据。
- 广播消息的本质是对多个 socket 分别 send。

### 阶段三：设计并实现协议

目标：让消息格式稳定，不依赖随意字符串。

完成内容：

- 定义消息类型。
- 每条消息带长度字段。
- 客户端和服务端都按同一协议编码、解码。

你要理解：

- TCP 粘包和半包为什么会发生。
- 长度头如何解决消息边界问题。
- JSON 适合做消息头，二进制适合传输文件内容。

### 阶段四：完成 Qt 图形界面

目标：做出可演示的聊天客户端。

完成内容：

- 登录窗口。
- 聊天窗口。
- 在线用户列表。
- 消息气泡或消息列表。
- 发送按钮和回车发送。

你要理解：

- Qt 信号槽如何把界面事件连接到代码逻辑。
- QTcpSocket 的 readyRead 信号如何处理服务器消息。
- 网络逻辑不要阻塞界面线程。

### 阶段五：完成文件传输

目标：实现项目的核心升级功能。

完成内容：

- 选择文件。
- 发送文件元信息。
- 分片发送文件内容。
- 接收端保存文件。
- 显示进度条。

你要理解：

- 文件为什么要分片传输。
- 如何根据已接收字节数计算进度。
- 文件消息和聊天消息为什么要用不同类型区分。

### 阶段六：完善和答辩准备

目标：让项目可展示、可解释、可维护。

完成内容：

- 整理代码结构。
- 增加异常处理。
- 编写 README。
- 准备演示步骤。
- 准备原理说明。

你要能回答：

- 为什么使用 TCP 而不是 UDP？
- 为什么服务端使用 epoll？
- 什么是粘包？你是怎么解决的？
- 文件传输为什么要分片？
- Qt 客户端如何接收服务器消息？
- 如果客户端突然断开，服务器如何处理？

## 9. 推荐实现顺序

建议不要一开始就做完整界面和文件传输。推荐顺序如下：

1. 命令行服务端 + 命令行客户端，实现单人连接。
2. 服务端支持多个客户端，完成群聊广播。
3. 加入协议封装，解决消息边界问题。
4. 使用 Qt 替换命令行客户端。
5. 加入在线用户列表。
6. 加入文件传输。
7. 做界面优化、异常处理和文档。

这样做的好处是每一步都有可运行结果，遇到问题时容易定位。
