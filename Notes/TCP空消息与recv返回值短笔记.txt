TCP 空消息与 recv 返回值短笔记

问题 1：如果客户端发了空消息，bytes_received == 0 吗？

答案：

不是。

recv() 返回 0 通常表示：

客户端正常关闭了连接。

也就是对方 close() 了 socket，或者程序退出了。

recv() 的返回值含义：

bytes_received > 0    收到了这么多字节
bytes_received == 0   客户端正常断开连接
bytes_received < 0    接收出错


问题 2：如果客户端发空消息，要特别处理吗？

答案：

聊天系统里通常要处理，但最好分两层：

客户端：

如果输入框为空，就不要发送。

服务端：

收到消息后也要做防御判断，避免恶意或异常客户端乱发数据。

注意：

TCP 层没有“空消息”概念，只有字节流。
“消息是否为空”应该由应用层判断。


问题 3：即使客户端发了空消息，bytes_received 也会 > 0 吗？

答案：

不一定，要看实际有没有发送字节。

情况 1：

send(sock, "", 0, 0);

发送 0 字节，基本等于什么都没发，服务端通常不会收到一条数据。

情况 2：

send(sock, "\n", 1, 0);

发送了一个换行符，服务端会收到 1 字节，所以 bytes_received == 1。

情况 3：

send(sock, " ", 1, 0);

发送了一个空格，服务端会收到 1 字节，所以 bytes_received == 1。

情况 4：

{"type":"chat","content":""}

聊天内容 content 是空的，但整条 JSON 协议本身有很多字符，所以 bytes_received 会 > 0。


核心总结：

bytes_received 只能说明有没有收到字节。
它不能直接说明聊天内容是不是空。

聊天内容是否为空，要在应用层解析后判断。
